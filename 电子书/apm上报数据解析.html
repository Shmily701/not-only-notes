    sogouAPM内存监控-线上内存溢出或内存泄漏数据监控反馈。

    内存

    实现原理

    sogouAPM的内存监控采用了快手开源的koom技术方案进行二次开发。堆栈数据采集使用APM中的业务监控方案上传服务端(key：leakTrace)

    监控时机

    监控工具：HandlerThread，默认5s进行一次轮训检查系统内存使用情况。开启条件需同时满足以下条件：

    1. 系统版本：大于5.0，小于10.0
    2. 手机存储空间大于5g
    3. 安装第一次启动后开始计时，时间不超过15天。
    4. 主进程

    dump时机

    dump是指将当前虚拟机使用内存的数据完整的copy一份出来。dump需要同时满足以下时机：

    1. 内存超过阈值（使用内存占比）
    业务组 业务组版本号 阈值（%）
    浏览器 6.2.50 10
    浏览器 6.3.0 10
    浏览器 6.3.5 40
    1. 超过阈值的次数大于设定值
    业务组 业务组版本号 超过次数
    浏览器 6.2.50 1
    浏览器 6.3.0 1
    1. 监控到有leak的对象

    上报条件

    上报条件也就是dump在内存中的对象满足上报条件时将会被记录，进一步上传服务器。

    • activity

    Activity成员变量：

    1. mOnDestroy == true

    或 

    1. mFinished == true
    • fragment
    1. mFragmentManager != null

    与

    1. mCalled != null && mCalled == true
    • bitmap
    1. mWidth != null && mHeight != null

    与

    （mWidth ✖️ mHeight）> （768✖️1366）

    • 大数组（基本类型）
    数据类型 阈值
    int 4✖️256✖️1024
    short 2✖️256✖️1024
    long 8✖️256✖️1024
    其他 同理

    以上述引用链为例，解释相关名称含义：

    • gcRoot：此引用链分析时，从哪种gcRoot开始分析，可不用过分关注。
    • instanceCount：泄漏此对象数量。
    • leakReason：泄漏原因。
      1. Activity泄漏：Activity Leak
      2. Bitmap泄漏：Bitmap Size
      3. Fragment泄漏：Fragment Leak
      4. Window泄漏：Window
      5. 大数组：primitive array size over
    • path：泄漏相关引用链
      1. declaredClass：类类型，（可为空）
      2. reference：类变量，换句话说就是对象持有者（不可空）
      3. referenceType：此节点类型。array-数组类型，INSTANCE_FIELD-成员变量，ARRAY_ENTRY-数组节点，STATIC_FIELD-静态变量，instance-泄漏对象。
        • signature:此引用链节点的签名。可不关注。

    上述虚拟机运行时环境参数含义解释：

    • analysisReason：触发时机，crash触发或者到达相关条件触发。
    • currentPage：当前上下文。

    • vss：虚拟耗用内存（包含共享库占用的内存），涉及内存映射，往往数据比较大。

    bitmap泄漏补充

    上报相关bitmap的堆栈，不一定说明此bitmap泄漏，只是说名此bitmap 超过了我们认为的最大值，可排查是否符合预期。

    大数组泄漏补充

    在Android8.0以前bitmap的绘制数据存储于Java层面，在mBuffer成员变量中，即有过多大数组问题都与此相关，排查上报bitmap是否符合预期。

    慢方法

    原理

    通过对接入方的方法进行插桩，并对主线程的handler进行监控，当方法执行时记录方法进入时间，方法退出时记录方法退出时间，求差，来完成对方法执行时间的统计。

    监控时机

    启动到退出

    上报时机

    方法退出时，将执行方法堆栈进行处理（计算、截取、统计），如果此方法超过设定阈值，那么上报，

    业务组 阈值（ms）
    浏览器 1000

    其他

    堆栈解释

    在方法堆栈上报处理过程中，方法堆栈会做简单裁剪，例如：get/set；
    一旦遇到耗时方法，后面的简单方法不与裁剪；
    如一个方法中有两个子方法，将保留其中一个最为耗时的子方法。
